隨著 0.17 版本的發佈，我們已經將注意力轉向接下來的工作。我們決定花一點時間規劃專案的走向，而不是直接投入到下一組功能的程式碼編寫中。

使用我們的 [Trello 白板](https://trello.com/b/R0O3CSrI/node-red-whiteboard) 來追蹤個別增強功能和更大的「史詩 (epics)」任務非常有助於展示我們心目中的計劃。但 Trello 板缺乏敘述性，無法將那些個別卡片描繪成一個整體的藍圖。

#### 關於版本號

我們也一直在思考版本號的問題。自專案開始以來，我們一直遵循語義化版本控制 (semantic versioning) 規則。版本號由三個部分組成：主版本號.次版本號.修訂號 (major.minor.patch)。何時增加每個部分的規則非常簡單：

 - 修訂號 (Patch) 在您進行向後相容的錯誤修復時增加
 - 次版本號 (Minor) 在您以向後相容的方式增加功能時增加
 - 主版本號 (Major) 在您進行不相容的 API 更改時增加。

這給了我們不必擔心 1.0 的藉口。我們一直專注於將每個版本作為對上一個版本的增量改進，竭盡全力確保不進行不相容的更改。以至於四年前從 0.3 版本導出的流程，仍然可以在今天的 0.17 版本中導入並使用。

但 1.0 版本具有特殊的意義。語義化版本 (SemVer) 規範是這樣說的：

> [我如何知道何時發佈 1.0.0？](http://semver.org/#how-do-i-know-when-to-release-100)

> 如果您的軟體已被用於生產環境，它可能應該已經是 1.0.0 了。如果您有一個使用者已經開始依賴的穩定 API，您應該已經是 1.0.0 了。如果您非常擔心向後相容性，您應該已經是 1.0.0 了。

今天有很多使用者在生產環境中使用 Node-RED —— 這是我們完全支持的。但我們也聽說有些使用者不願進入生產環境，因為我們還沒有傳說中的 1.0 版本號。

所以，是時候談談 1.0 了。

#### 邁向 1.0

我們一直在問自己的問題是：要達到 1.0 版本需要做什麼？白板上的哪些項目我們想優先完成，順序又是如何？

為了幫助解決這個問題，我們確定了許多我們想要解決的高層級主題：

 - 工作流程 (Workflow) —— 個別開發者如何提高生產力
 - 協作 (Collaboration) —— 在開發團隊中工作
 - 可擴充性 (Extensibility) —— 提供正確的擴充點和 API 以適配 Node-RED
 - 擴展性 (Scaling) —— 讓建立能夠擴展以處理日益增長工作負載的流程變得更容易
 - API 穩定性 (API Stability) —— 提供一個可以長期開發的穩定基礎

在這些主題的背景下思考白板上的功能，引導我們制定了這個路線圖。大多數個別項目對於那些關注 Slack 或郵件列表對話的人來說應該很熟悉，但其中有幾個尚未被廣泛討論 —— 例如專案 (Projects) 和新的節點訊息傳遞 API (Node Messaging API)。

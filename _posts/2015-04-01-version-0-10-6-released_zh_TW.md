Node-RED 0.10.6 現在已經可以 [下載](https://github.com/node-red/node-red/archive/0.10.6.zip) 或透過 [npm 安裝](https://npmjs.org/package/node-red)。

如果升級，請閱讀 [升級說明](/docs/getting-started/upgrading_zh_TW.html)。

請記住，目前我們 **不支持** Node.js v0.12 或 io.js。

### 關於版本號

您可能已經注意到這次發佈的版本是 `0.10.6`。您可能想知道 `0.10.5` 發生了什麼。這標誌著我們將採取略有不同的版本號編寫方式的首個版本。

**簡而言之**：發佈到 npm 的正式版本最後一位數字將為偶數。GitHub 上正在開發的最新程式碼將使用下一個奇數。

以前，我在發佈新版本之前做的最後一次提交是增加 `package.json` 中的版本號。在那之後，任何從 npm 安裝版本的人都會獲得該版本。然後我們會在 git 中繼續開發。如果有人想通過 clone 儲存庫來嘗試 git 中的最新程式碼，它在 `package.json` 中的版本號仍然會與上一個版本相同。

這在使用者報告問題時引起了一些困惑，也為將 Node-RED 作為依賴項引入的自動構建帶來了一些技術麻煩。

通過在發佈後立即增加 git 中的版本（使其變為奇數），並在下一次發佈前立即增加它（使其變為偶數），有助於識別不同的版本層級。

### 編輯器變更

編輯器進行了許多小的變更。值得強調的一些變更有：

 - 我們對編輯器中繪製流程的方式進行了一些性能改進。您的節點/連線越多，就越能感覺到改進。

 - 調色板 (palette) 進行了遲來的更新，使節點的外觀與它們在工作區中的樣子保持一致。

 ![調色板更新](/blog/content/images/2015/Mar/Selection_164.png)

 - 如果您嘗試在有未部署更改的情況下離開編輯器，您將收到瀏覽器警告，提示您將丟失更改，並為您提供留在頁面上的選項。

 ![確認導航](/blog/content/images/2015/Mar/Selection_165.png)


 - 一個常見的問題是 Inject 節點的按鈕在部署前被點擊，這是不起作用的。或者當它有未部署的更改時，點擊按鈕會導致觸發當前的部署版本。

 現在，如果節點有未部署的更改，按鈕在視覺上會被禁用。

 ![Inject 按鈕](/blog/content/images/2015/Mar/Selection_166.png)

 - 另一個節點動作按鈕可能導致意外行為的區域是在編輯子流程 (subflow) 時。子流程最好被視為節點的模板。編輯子流程時，您是在編輯模板，而不是子流程的實際實例。在此視圖中，節點動作按鈕（如 Inject 節點上的按鈕）不對應運行時中的單個節點，因此點擊它們將不起作用。為了反映這一點，在編輯子流程時按鈕會被禁用。

 - 提到子流程，您現在可以通過雙擊調色板中的條目來進入子流程的編輯視圖。

 - 我們已從 Eclipse Orion 富文本編輯器組件轉向在 Function、Template 和 Comment 節點中使用 [ACE 程式碼編輯器](http://ace.c9.io/)。我們仍然在編輯器中載入 Orion 庫，因為我們知道有第三方節點依賴它。

### 節點變更

#### Catch 節點

調色板中的許多節點在行為上都是樂觀的；它們假設事情會成功，因此忽略了錯誤會導致流程停止的事實。

在現實中，事情確實會出錯。文件寫入失敗、TCP 套接字連接失敗、使用者編寫的 Function 節點包含錯誤。一旦您想建立一個可以處理這些場景的流程，就會變得有點棘手。

此版本引入了新的 Catch 節點。此節點可用於捕獲同一標籤頁上其他節點拋出的錯誤，並觸發錯誤處理流程。該節點會發送出錯時正在處理的訊息，並允許執行所需的任何步驟。

對於基於 HTTP 請求的流程，這尤為重要，因為無論發生什麼，流程都必須完成並發送回應。

![錯誤流程範例](/blog/content/images/2015/Mar/Selection_168.png)

我們將改進各個節點的文件，以強調哪些節點會在什麼情況下觸發 Catch 節點。

#### Function 節點

Function 節點有兩個新功能。

首先，如果它需要將內容記錄到 Node-RED 日誌中，它現在可以使用以下函數：

    node.log("some logging message");
    node.warn("this is a warning");
    node.error("something went wrong",msg);

最後一個 `node.error` 可以接受一個可選的第二個參數，即正在處理的訊息。如果傳入該訊息，它將觸發同一標籤頁上的任何 Catch 節點。

第二個變化是人們多次要求的，但花了一點時間才確保我們正確地接入了。

與其要求 Function 節點返回要發送的訊息，它現在也可以隨時調用新的 `node.send()` 函數來發送訊息。這一小小的增加意味著可以在 Function 節點中編寫執行非同步操作的程式碼。

增加這一功能的後果是，在某些情況下，Function 需要在節點停止時獲得通知，以便它可以取消任何待處理的活動。就像運行時節點 API 一樣，Function 可以通過調用以下代碼為 `close` 事件註冊自己的處理程序：

    node.on('close',function() { })

[文件](/docs/writing-functions_zh_TW.html) 已更新以涵蓋這些內容。



#### Change 節點

Change 節點使設置、更改和刪除訊息屬性變得容易，而無需訴諸 Function 節點。使用者仍然訴諸 Function 節點的一個常見原因是他們想要進行多次更改，而在流程中為每個所需的更改設置一個 Change 節點會使工作區顯得擁擠。

此版本引入了更新後的 Change 節點，允許您指定應用於通過的每條訊息的多條規則。

![Change 節點編輯對話框](/blog/content/images/2015/Mar/Selection_169.png)


### 運行時變更

正如在過去幾個版本中所強調的，我們已經棄用了某些節點中的行為，即訊息的屬性可以覆蓋節點中明確設置的屬性。我們在 0.10.1 版本中增加了警告，以提醒使用者他們正在使用這種已棄用的行為。

在此版本中，我們現在切換了節點實現，使得訊息的屬性不會覆蓋節點中明確設置的屬性。我們更新了棄用警告，如果我們發現訊息中的屬性在以前會覆蓋節點屬性，則會記錄該警告。

### 節點 API 變更

我們在 [節點定義](/docs/creating-nodes/node-html_zh_TW.html#node-definition) 中增加了 `onpaletteadd` 和 `onpaletteremove` 屬性。如果節點對 UI 做了任何貢獻，它應該在 `onpaletteadd` 函數中完成，並在 `onpaletteremove` 函數中移除。這確保了節點可以動態地從調色板中添加和移除，而不會留下殘餘。

節點現在可以通過多次調用 `this.on('close',function(){...})` 來定義多個 `close` 事件的處理程序。

### 命令行管理工具

我們還發佈了 Node-RED 命令行管理工具。這允許您遠程添加/刪除/啟用/禁用 Node-RED 中的節點，而無需重啟。

更多細節在 [文件](/docs/node-red-admin_zh_TW.html) 中。

此工具使用了我們已經實施了幾個版本的管理 API。隨著該工具的發佈，我們現在很高興該 API 足夠穩定，供他人使用。該 API 在我們文件的 [新 API 章節](/docs/api/index_zh_TW.html) 中有記錄。

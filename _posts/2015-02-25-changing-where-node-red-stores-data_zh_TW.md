我們最近一直在思考的一件事是我們的升級流程。
使用者如何以盡可能少的努力從 Node-RED 的版本 A 過渡到版本 B。

預設情況下，我們會將您的數據寫入 Node-RED 的安裝目錄。這意味著您不能直接用最新版本的 Node-RED 替換該目錄。您必須首先備份您的數據。雖然這顯然總是一件審慎的事情，但具體構成「您的數據」的文件/目錄列表很長，很容易出錯導致某些東西丟失 —— 特別是對於不習慣複製文件等操作的使用者。

這也意味著我們經常告訴使用者從儲存庫的 git clone 運行 —— 因為那裡的升級路徑只是運行 `git pull` 和 `npm update`。但這讓使用者運行的是最新的開發程式碼，而不是最新的穩定版本 —— 由於我稍後會討論的原因，這遠非理想。

我們已經有了 `userDir` 設置一段時間了，它允許您指向一個應該放置所有數據的目錄 —— 這有助於將其與 Node-RED 安裝分開。我們關於「正常運作 (Just Working)」的規則之一是我們應該有正確的預設行為。一旦我們要求使用者編輯 settings.js 以確保他們的數據放在合理的地方，我們就失敗了。

### 一個新的預設值

即將發佈的 Node-RED 0.10.4 版本將包含對 Node-RED 預設行為的更改：使用者數據將被寫入目錄 `$HOME/.node-red/`。這確保了您的數據與 Node-RED 安裝保持分離，這意味著升級更容易完成 —— 您只需升級 Node-RED。


為了確保完全的向後兼容性，Node-RED 在決定存儲數據的位置時會採取以下步驟：

 1. 如果在啟動時明確指定了 `userDir`，無論是通過 `settings.js`（如前所述），還是通過新的命令行參數：`node red.js --userDir /var/node-red/data`

  這意味著已經設置了 `userDir` 的使用者不受影響，而且如果您想在配置之間快速切換，明確指向特定目錄會更容易。

 2. 如果我們在 Node-RED 安裝目錄中檢測到使用者數據的存在（特別是 `.config.json` 文件的存在），我們將繼續使用 Node-RED 安裝目錄來存儲數據。

  這將意味著現有的 Node-RED 安裝將像以前一樣運行。我們不會將現有數據移動到 `$HOME/.node-red`

 3. 如果以上都不適用，我們將使用 `$HOME/.node-red`。換句話說，全新的安裝會這樣做。

關於 [如何升級](/docs/getting-started/upgrading_zh_TW.html) 的文件將更新，包括如何手動將您的數據遷移到 `$HOME/.node-red` 的指南。


## 安裝 Node-RED


這一變化也帶來了我們推薦安裝 Node-RED 方式的變化。與其下載 zip 文件，我們將推薦使用 npm 來安裝 Node-RED。

以下命令將執行 Node-RED 的全域安裝：

    sudo npm install -g node-red

這還會將新命令 `node-red` 放到您的路徑 (path) 中，允許您從任何目錄運行。

當有新版本可用時，您只需運行以下命令即可完成安裝：

    sudo npm update -g node-red


如果您選擇在本地安裝而不是全域安裝，您仍然可以使用 `npm update` 命令進行升級，但您不會在路徑中獲得 `node-red` 命令。


## 從 git 運行

一些使用者選擇 clone git 儲存庫並從那裡運行。這具有獲得所有最新變化的優點，但始終存在獲取尚在開發中程式碼的風險。

在不久的將來，我們將在開發過程中增加一個構建 (build) 步驟，它將介於 git 中的程式碼和構成 Node-RED 可運行實例的內容之間。

如果不首先運行構建，您將 *不能* 直接從 git clone 運行 Node-RED。這將需要安裝開發依賴項 —— 而不僅僅是生產依賴項。

我們將在它進入儲存庫之前分享更多細節。但作為提前警告，如果您是為了獲得最新程式碼而從 git 運行，請做好準備。

如果您是為了方便在版本之間升級而從 git 運行，那麼在我們發佈 0.10.4 之後，您應該轉向上述的 `npm install -g` 方法。

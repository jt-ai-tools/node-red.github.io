### Node.js 版本支援

Node-RED 0.20 需要 Node.js 8.x 或 10.x — 我們不再支援在任何更早的版本上執行。

### 執行環境/編輯器分離

0.20 的程式碼庫進行了重大重構，現在已拆分為多個模組。雖然這對大多數終端使用者來說意義不大，但這是邁向 1.0 版的重要一步。

這項工作的一部分包括為各種模組及其公開的 API 編寫文件。現在可以在[這裡](https://nodered.org/docs/api/modules/v/0.20.0/index_zh_TW.html)查看。

關於如何單獨使用這些模組，還有很多需要記錄的地方。如果您有興趣了解這方面的內容，請與我們聯絡。

### 子流程增強

#### 子流程狀態

現在有一個新選項可以為子流程新增「狀態（status）」輸出。這可以用於更新子流程實例節點的狀態。目前在子流程中，如果子流程內部的任何節點發出狀態事件，它會傳遞到包含該子流程實例的流程 — 但它看起來不像是*來自*該子流程實例。現在，如果使用了這個新的狀態輸出，則只有傳遞給它的訊息才會傳遞給父流程，並且它們*將*被正確地識別為來自子流程。這在處理狀態事件和子流程時提供了更多的控制和靈活性。

#### 存取父流程 Context

在子流程中使用 context 時，現在可以存取子流程實例所在流程的 context。這是透過在 context 鍵名前加上 `$parent.` 來實現的。例如，如果一個流程有一個名為 `foo` 的 context 值，則子流程可以使用 `flow.get("$parent.foo");` 來存取它。

#### 子流程實例屬性

現在可以在子流程中定義環境變數。這些是子流程中的任何節點都可以引用的值，引用方式與引用普通環境變數相同：

 - 將節點屬性設置為 `$(ENV_VAR_NAME)`
 - 在 typedInput 中選擇 `env` 類型
 - 在 Function 節點中使用 `env.get('ENV_VAR_NAME')`

這些屬性可以在子流程範本上設置，也可以在子流程的各個實例上設置。這意味著您可以設計一個可以針對每個實例進行自定義的子流程。與只能是字串的行程級（process-wide）環境變數不同，子流程上定義的環境變數可以是多種類型 — 字串、數字、布林值、JSON、Buffer，甚至是對另一個環境變數的引用。最後一個選項意味著您可以建立由其他環境變數組成的複合環境變數……噢，想想其中的可能性吧。

當流程引用環境變數時，如果它未在子流程（或包含該子流程的子流程，或包含*該*子流程的子流程……依此類推）中定義，它將回退到行程級環境變數。如果那裡也沒有設置，它將返回空字串 `""` — 而不是 `undefined`。

這些新的子流程功能解決了我們收到的大量常見請求和反饋。我們認為這將使它們成為 Node-RED 中更強大的工具。



### 快速新增對話框

- 編輯器中的快速新增模式 (Ctrl/Cmd-點擊) 進行了幾次更新。現在在對話框開啟時會顯示一個佔位節點。如果您使用 Ctrl/Cmd-Enter 確認要新增的節點，它會新增該節點並保持對話框開啟，以便您可以新增另一個節點。這意味著您可以完全從鍵盤快速新增一連串節點（在初始 Ctrl-點擊開啟對話框之後）。

![](/blog/content/images/2019/03/quick-add.gif)

### 連接連結節點

當您選擇一個 Link 節點時，它現在會顯示一個額外的連接埠，您可以從中點擊並拖動一條線到另一個 Link 節點以將它們連接起來。與以前一樣，這些連結線僅在您選擇了連結節點時才會顯示。

![](/blog/content/images/2019/03/link-node-wiring.gif)


### 效能改進

在 0.19 發生效能倒退之後，Change、Switch 和 Join 節點進行了一些效能改進 — 特別是如果它們與 context 交互或使用 JSONata 表達式。

例如，給定一個使用 JSONata 計算 `payload*2` 的 Change 節點，在我的筆記型電腦上，在 0.19.5 中處理 6.5 萬條訊息大約需要 12 秒（5000 條訊息/秒）。在此版本中，現在大約需要 5 秒（1.2 萬條訊息/秒）。

### 節點編輯對話框變更

節點的編輯對話框已重構為一組分頁標籤，而不是可摺疊區塊。這意味著我們可以在不損失寶貴垂直空間的情況下向對話框新增新區塊。

#### 節點說明

其中一個分頁標籤是說明欄位，因此您可以為*任何*節點新增 Markdown 格式的說明。當您選擇該節點時，內容將顯示在「資訊」側邊欄中。

為了幫助進行 Markdown 格式設置，對話框中新增了一個新的工具列。它還可以展開為帶有格式化文字預覽的全螢幕編輯器。

#### 節點圖示

在 0.19 中，我們引入了自定義節點圖示的能力。這已擴展到包括從 Font-Awesome v4 提供的圖示範圍中進行選擇（不包括任何特定品牌的圖示）。


### 使用分頁標籤

可以透過雙擊標籤列來新增新的流程分頁標籤。標籤將插入到您點擊的位置 — 例如在兩個現有流程之間。

Cmd (或 Ctrl) - 點擊標籤可以選中它。您可以選擇多個標籤。選中後，按 delete 鍵將刪除標籤。

開啟「匯出到剪貼簿」對話框（從功能表或 Cmd/Ctrl-E）將允許您匯出選中的標籤。


### 編輯器

 - 當您將選中的節點轉換為子流程時，這些節點將移回到子流程標籤的左上角。
 - 您可以在 Context 側邊欄中刪除單個 context 條目。
 - 您可以在配置節點（Config node）側邊欄中選擇配置節點，Ctrl-點擊可選擇多個。按 delete 鍵將刪除選中的配置節點 — 這比雙擊開啟每個節點的對話框要容易得多。
 - 當您在專案側邊欄中啟動 git push/pull，或從「管理調色盤」對話框安裝/移除節點時，您會看到一個「查看日誌」按鈕。這將開啟新的事件日誌（Event Log，也可以從功能表「檢視」->「事件日誌」存取）。這讓您可以查看這些操作執行的底層 git/npm 命令的更多日誌輸出。
 - 「從剪貼簿匯入」對話框允許您上傳檔案，而不是從剪貼簿貼上。如果您提供了無效的 JSON，它在突出顯示錯誤方面也做得更好。
 - 「匯出到剪貼簿」對話框允許您將流程下載為檔案，而不是複製到剪貼簿。
 - 部署功能表中新增了一個新的「重啟流程」選項 — 這會導致執行環境從儲存空間重新載入其活動流程並重啟它們。
 - 如果某個模組提供的節點類型已經安裝，「管理調色盤」對話框將拒絕安裝該模組。

### 執行環境

 - 新增 `--safe` 模式標誌，允許在不執行流程的情況下啟動。如果某個流程導致 Node-RED 崩潰，可以使用此模式 — 在安全模式下啟動將允許使用編輯器修改流程，然後部署更改以使其退出安全模式。
 - 為自動存取 adminAPI 新增了設定定義的 `accessToken`。
 - 允許從命令列載入專案 — 在提供流程檔案名稱的地方指定專案名稱將載入該專案。
 - 新增了一個設定 `exportGlobalContextKeys`，可用於防止由 `functionGlobalContext` 設置的全域 context 值顯示在 context 資料側邊欄中。執行環境設定檔案中提供的預設值為 `false`，因此新安裝的版本將不會在側邊欄中看到這些值。沒有此設定的現有安裝版本仍將看到它們。

### 節點

 - Catch 節點有一個新選項，僅處理尚未被另一個 catch 節點擷取的節點錯誤。這意味著您可以讓 catch 節點針對流程中的特定節點，然後再讓一個 catch 節點擷取其餘的所有內容。
  - HTTP Request 節點現在可以選擇自動將 `msg.payload` 編碼為 GET 請求的查詢字串參數。它還有一個新選項可以對 cookie 進行 URL 編碼（如果在極少數情況下，它們需要編碼而不是保留為純文字）。此節點現在還支援 Bearer 和 Digest 認證模式。
  - MQTT In 節點現在可以配置為假設它接收到的訊息是 JSON 並發出解析後的物件。如果您確定接收到的所有內容都將是有效的 JSON，這將省去在 MQTT In 節點之後新增 JSON 節點的麻煩。如果節點接收到無效的 JSON，它將記錄一個錯誤，可以使用 Catch 節點進行處理。
  - TCP/Websocket 節點在客戶端連接和斷開時發出的狀態訊息現在包含有關連接/斷開客戶端的更多資訊。

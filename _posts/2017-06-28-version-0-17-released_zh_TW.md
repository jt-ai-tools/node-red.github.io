Node-RED 0.17 現在已經可以 [下載](https://github.com/node-red/node-red/releases/download/0.17.0/node-red-0.17.0.zip) 或透過 [npm 安裝](https://npmjs.org/package/node-red)。

如果升級，請閱讀 [升級說明](/docs/getting-started/upgrading_zh_TW.html)。

對於 Raspberry Pi，0.17 將不會透過標準的 Raspberry Pi Jessie 儲存庫提供。如果您仍在使用預裝版本，請參閱 [Raspberry Pi 文件](/docs/hardware/raspberrypi_zh_TW.html#upgrading) 了解如何升級。

---

這個版本製作了一段時間，但為您帶來了許多實用的新功能。整個程式碼庫有超過 300 次提交，創紀錄地有 20 位不同的貢獻者。


### 簡要摘要 (TL;DR)

 - Debug 面板改進 —— 複製訊息路徑/內容以及更好的過濾
 - 節點端口上的工具提示標籤
 - 節點說明文字的新風格指南
 - 更好的 JSON、Buffer 和 JSONata 表達式編輯工具
 - 新的設置面板，匯集了編輯器的自定義選項...
 - ... 包括自定義鍵盤快捷鍵
 - 透過調色板管理員更新節點
 - Split/Join 節點的更多選項
 - HTTP In 節點中的檔案上傳及其他更新
 - 支持 OAuth/OpenID 方案以保護編輯器
 - 許多其他您應該在下面閱讀的內容！

### 編輯器更新

#### 更強大的 Debug 面板

在我們上次對 Debug 側邊欄所做的改進基礎上，我們繼續使其更加實用。

![](/blog/content/images/2017/06/debug-tools.png)

當您將游標懸停在偵錯訊息的任何元素上時，會出現一個迷你工具列，提供一些操作。第一個按鈕將複製該訊息元素的路徑到剪貼簿。例如，在上面的螢幕截圖中，它將複製 `payload["First Name"]`。這可以貼到您可能需要引用該元素的任何地方，例如在 Change 節點中。

第二個按鈕將複製該元素的值 —— 同樣，在這個例子中，您會得到 `Fred`。如果該元素是對象或數組，您將獲得 JSON 編碼版本。

第三個按鈕讓您「釘選」這個訊息元素。當來自同一個 Debug 節點的另一條訊息到達時，它將自動展開以顯示所有釘選的元素。

過濾選項也得到了擴展。除了現有的根據您正在查看的流程進行過濾的選項外，您還可以過濾視圖以顯示特定的節點。

![](/blog/content/images/2017/06/debug-filter.png)

#### 端口標籤 (Port Labels)

現在所有的節點端口在滑鼠懸停時都可以顯示工具提示。一個節點可以在其定義中定義自己的標籤，或者使用者可以指定自定義標籤。例如，Switch 節點將自動建立標籤，告訴您每個輸出適用什麼規則。

![](/blog/content/images/2017/06/switch-labels.gif)

節點編輯器有一個新的「端口標籤」部分，您可以在其中為任何節點提供自己的標籤。

![](/blog/content/images/2017/06/switch-labels-edit.gif)



#### 節點資訊面板

資訊側邊欄進行了通盤整理，我們為節點說明文字引入了 [風格指南](/docs/creating-nodes/help-style-guide_zh_TW.html)。為節點提供的資訊建立更標準的結構，將使使用者更容易理解如何使用任何特定節點。

如果您已經發佈了自己的節點，我們強烈建議您更新它們以遵循風格指南。

#### 表達式/JSON/Buffer 編輯器

我們提供的 `typedInput` 通用組件已擴展以支援更多類型，並使處理某些現有類型變得更容易。

我們增加了一個 Buffer 類型，以便您可以處理 node.js Buffer 對象。您可以將值輸入為 JSON 字節數組，例如 `[1,2,3,4]`，運行時會將其轉換為相應的 Buffer 對象。

`typedInput` 的單行設計不適合處理較長的內容，如 JSON 或較長的 JSONata 表達式。對於這些類型以及新的 Buffer 類型，我們增加了透過點擊輸入框右側的按鈕將 `typedInput` 展開為更大的編輯器的功能，以提供更多空間。

JSONata 編輯器提供了一個標籤頁來在編輯器內測試您的表達式。您可以貼入一個範例訊息（如上所述，您現在可以直接從 Debug 側邊欄複製它）並查看表達式的結果。它還包含 JSONata 語言的完整函數參考。

說到 JSONata，我們也更新到了最新版本 (1.2.6)，這帶來了許多新函數，包括 `$flowContext` 和 `$globalContext`，可用於在表達式中從上下文中檢索值。我們還做到了您不再需要在表達式中使用 `msg.` 前綴 —— 因此以前您會使用 `msg.payload` 的地方，現在只需使用 `payload`。現有的表達式將繼續有效 —— 但我們建議更新它們。

Buffer 編輯器讓您輸入 JSON 數組，並在下方以十六進位顯示數據。如果您輸入了無效的 JSON，它將其視為原始 UTF-8 數據並將其轉換為其 Buffer 表示形式。

#### 設置面板

我們在編輯器中引入了一個新的設置面板，匯集了您可以用來定製編輯器的所有選項。

這是您現在可以找到以前在「檢視 (view)」選單下的一些選項的地方，如網格選項。這也是您可以找到鍵盤快捷鍵列表的地方 —— 包括根據自己的喜好自定義任何快捷鍵的功能。請注意，這些選項存儲在瀏覽器中 —— 因此如果您從不同的瀏覽器存取編輯器，它們將不會被保存。

#### 調色板管理員更新

隨著新設置面板的就緒，調色板管理員視圖也被移了過來。

節點視圖已更新，以突出顯示任何有可用更新的節點，並讓您啟動安裝。它還警告您 *必須* 手動重新啟動 Node-RED 才能完成更新。


### 節點更新

我之前提到我們為節點幫助增加了風格指南。大多數核心節點已更新以使用該指南。

一些節點對其編輯對話框進行了微調，以提高其可用性 —— 尤其是關於解析器節點 (JSON, HTML, CSV 等)。

#### Split/Join

Split 節點已更新，提供了關於應如何處理拆分不同訊息類型的更多選項。例如，數組現在可以拆分為固定長度的子數組，而不是單個元素，它現在還可以處理 Buffer payload。

增加了一種新的「串流 (streaming)」模式，可用於重新對齊訊息流，以確保每條訊息都是一條「完整」的訊息。例如，如果數據源正在發送由換行符分隔的數據包，但它無法確保數據包與單個訊息對齊，您可能會收到包含以下內容的兩條訊息：

```
"one\ntwo\nthr" "ee\nfour\n"
```

注意包含 `three` 的數據包被拆分到了兩條訊息中。在串流模式下，Split 節點將能夠將其轉換為包含以下內容的流：

```
"one" "two" "three" "four"
```

#### HTTP 節點

HTTP 節點有一些實用的更新。

HTTP In 節點現在支持檔案上傳。必須在節點中啟用（且預設不啟用）。啟用後，請求中包含的任何檔案都將在 `msg.req.files` 對象下提供。請注意，檔案內容包含在訊息中，而不是寫入臨時文件。

HTTP Response 節點現在讓您在節點內硬編碼響應狀態碼和 HTTP 標頭，作為在每條訊息上手動設置它們的替代方案。

HTTP Request 節點現在具有更好的 cookie 處理功能 —— 使在請求上設置 cookie 並接收響應中設置的 cookie 變得更容易。請注意，該節點會自動跟隨 HTTP 重定向，並且只會返回其在重定向序列中所做的最後一次請求上設置的 cookie。對於某些登入 API，這意味著您無法輕鬆存取序列早期產生的 cookie。這是該節點的一個顯著限制，我們希望解決它 —— 但這樣做將涉及對其內部結構進行重大更改。與其倉促將其加入此版本，我們決定將其延後到下一個版本，以便我們可以花時間確保在此過程中不會發生任何破壞。

最後，我們讓 HTTP Request 節點的鏈接變得更容易。到目前為止，這比應有的情況要困難。Request 節點將其收到的標頭作為 `msg.headers` 發送。它也使用 `msg.headers` 來設置它所發出請求的標頭。這意味著給定兩個連續的 Request 節點，第一個返回的標頭會修改第二個所做的請求。這很少是正確的操作，並可能導致各種副作用。解決方法是在節點之間明確刪除 `msg.headers` —— 這是您必須知道的事情。在此版本中，我們現在會檢測一條請求節點的標頭何時被傳遞給另一個，並且除非它們以某種方式被有目的地修改過，否則第二個節點現在將忽略它們。


#### 其他節點

一如既往，其他節點也有一系列較小的更新。

 - Delay 節點現在將接受 `msg.delay` 來設置每條訊息的延遲間隔。
 - Exec 節點現在將接受 `msg.kill` 以殺死它之前啟動的進程。
 - File 節點現在可以為檔案內容發送一系列訊息。這些訊息使用與 Split/Join 節點相同的 `msg.parts` 屬性，因此可以饋入 Join 節點以恢復為單個訊息。
 - Template 可以發送解析後的 JSON 而不是原始文字。
 - TLS 配置節點現在允許在編輯器中上傳所需的證書文件，而不是指向本地文件。


### 運行時更新

#### 抑制捕獲到的錯誤

如果一個節點記錄了錯誤並提供了一個訊息對象，它可以觸發流程中的任何 Catch 節點。該機制已經實施一段時間了。

在大多數程式語言中，如果您捕獲了一個錯誤，就由您決定是否需要對其進行記錄。我們已將我們的行為調整為與該方法一致，因此現在我們將僅記錄那些未被 Catch 節點捕獲的錯誤。

#### 支持管理介面的其他身份驗證方案

我們擴展了編輯器支持的身份驗證機制，納入了 [Passport](http://passportjs.org/) 提供的許多機制。更具體地說，這意味著您現在可以使用 OAuth 和 OpenID 身份提供商來保護您的編輯器。

我們已經為 [Twitter](https://github.com/node-red/node-red-auth-twitter) 和 [GitHub](https://github.com/node-red/node-red-auth-github) 發佈了範例供您開始。

*注意*：我們尚未更新 `node-red-admin` 工具以支援這些身份驗證方案。

#### 節點 API 變更

`context` 對象有一個新函數 `context.keys()`，它返回存儲在上下文中的所有鍵的數組。這適用於節點自身的上下文以及流程和全域上下文對象。

當部署新流程時，所有節點在重新啟動之前會先停止（或 `closed`）。節點可以註冊的 `close` 事件處理程序現在可以接收一個布林標誌，以告知節點它是否正在被移除而不僅僅是重啟。

關閉節點的調用現在也會在 15 秒後超時。這將防止行為不當的節點掛起整個運行時。*注意*：掛起被視為節點的錯誤，任何此類情況都應作為問題向該節點提出。15 秒的超時時間可以使用設置文件中的 `nodeCloseTimeout` 屬性進行更改。

隨著 Node-RED 1.0 中向非同步訊息傳遞的轉變，我們也在更改某些訊息在節點之間複製的方式。此領域的行為以前並不總是清晰，並且可能會給不熟悉 JavaScript 物件處理某些原則的終端使用者帶來意想不到的結果。

這篇部落格文章解釋了什麼是訊息複製、為什麼它是必要的，以及 1.0 中有哪些變化。


### 傳址 (Pass-by-reference)

在深入研究訊息複製的細節之前，我們再次繞道了解一下 JavaScript 的運作方式。

讓我們考慮以下程式碼：

```javascript
> let a = { payload: "hello" };
> let b = a;
> b.payload = "goodbye";
> console.log(a)
{ payload: 'goodbye' }
```

我們建立一個新物件並將其分配給變數 `a`。然後我們將變數 `b` 分配為 `a` 的值。接下來我們更改 `b.payload` 的值。最後，我們印出原始變數 `a`。

就像變魔術一樣，我們對變數 `b` 所做的更改也同樣發生在變數 `a` 上。這是因為我們*沒有*製作物件的副本 — 我們在記憶體中建立了指向同一個物件的新引用。

這被稱為「傳址」，如果您對此沒有準備，可能會導致意想不到的結果。

### 訊息複製

在 Node-RED 流程中，當一個節點接收到一條訊息（這是一個 JavaScript 物件）時，它可以隨意修改該訊息，然後將其傳遞給它連接的任何節點。

![](/blog/content/images/2019/09/flow1.svg)

當節點 A 發送訊息時，它會生成兩個「發送事件」 — 一個給節點 B，一個給節點 C。如果我們只是將訊息傳遞給節點 B 然後再傳遞給節點 C，那麼節點 B 對訊息所做的任何修改對節點 C 都是可見的。

這就是需要複製訊息的地方。Node-RED 在訊息傳遞之前會自動複製訊息，以防止這種類型的跨分支修改。

但事情並非如此簡單。複製訊息的成本可能很高。對於沒有分支的單行節點流程，通常不需要對訊息進行任何複製。

![](/blog/content/images/2019/09/flow2.svg)

因此，程式碼會嘗試最佳化何時複製訊息。演算法如下：

> 當呼叫 `node.send()` 時，它會生成一個發送事件列表。第一個發送事件直接使用給定的訊息物件。其餘所有的發送事件在傳遞之前都會先複製訊息。

基本上這意味著，對於連接到另一個節點的節點，呼叫 `node.send(msg)` 將*不會*複製該訊息，因為沒有必要。

但這個演算法也有其局限性。特別是，我們無法處理 Function 節點使用*同一個*訊息物件多次呼叫 `node.send()` 的情況。

例如，考慮來自 Function 節點的以下程式碼：

```javascript
msg.topic = "A";
msg.payload = "1";
node.send(msg);

msg.topic = "B";
msg.payload = "2";
node.send(msg);
```

每次單獨呼叫 `node.send()` 都會生成一個發送事件，因此訊息不會被複製。

對於*某些*流程，在 Node-RED 1.0 之前這不是問題。如果流程中的後續節點完全是同步的，則第一次 `node.send()` 呼叫將在整個流程中傳遞並完成，然後執行權才會返回到該函數並為第二次呼叫修改訊息。

但隨著 Node-RED 1.0 引入完全非同步的訊息傳遞，這種程式碼模式可能會變得不安全。訊息可能會在第一次 `node.send()` 呼叫的發送事件送達*之前*被修改。

這可能會導致難以察覺的細微問題。沒有巨大的聲響來提醒使用者出現了問題。如果上面的程式碼連接到 MQTT 節點，它將在主題 `B` 上生成重複訊息，而在主題 `A` 上則什麼也沒有。

我們正在更改 Function 節點預設使用的複製方法，以防止此問題。

### 預設進行複製

在 Node-RED 1.0 中，當 Function 節點呼叫 `node.send()` 時，它現在將複製每一條訊息，包括第一條。這將確保像上面那樣的程式碼能夠繼續運作。

但遺憾的是，這並非沒有代價。複製*可能*是一項昂貴的操作。對於許多使用者來說，這根本不重要 — 他們的訊息相對較小且頻率較低。

對於具有非常大的訊息、高訊息速率，以及更關鍵的是使用*無法複製*的訊息的流程（例如，以高速率傳輸視訊訊框的流程），這將是一個更大的問題。

對於這些流程，我們在 `node.send()` 中引入了一個新的選擇性參數，它將保留現有的行為：

```javascript
node.send(msg, false);
```

這將告訴 Function 節點*不要*複製訊息 — 儘管規則仍然適用，即如果流程產生分支，則第二個及之後的分支*仍將*獲得複製的訊息。

### 為什麼要更改預設行為？

對於我們所做的每一次更改，我們都必須評估它可能產生的影響。目標是最小化這種影響，並保持流程按使用者預期運作。

當這個問題出現時，有兩種可能的處理方法。

一種選擇是不更改程式碼，並更新我們的文件，解釋您真的不應該重複使用訊息物件，並應使用 `RED.util.cloneMessage()` 在 Function 程式碼中手動複製訊息。

這種方法的問題在於，它讓使用者不確定應該為*他們的*流程做什麼。我們擁有龐大的使用者群，他們不是經驗豐富的 JavaScript 開發者，對 Node-RED 的內部運作也不感興趣。他們期望自己的流程能正常運作。由於這導致的任何問題都是細微且難以追蹤的，因此潛在影響非常高。

另一種選擇，也是我們選擇採取的，是更改預設行為，以確保大多數使用者的流程能繼續運作，而無需更改*任何內容*。

此選項的一個缺點是，會有一些流程依賴於非複製行為。然而，它們現在會以一種更明顯的方式失敗；在通過的第一條訊息時複製就會失敗。這將使識別出故障的 Function 節點並根據需要新增 `false` 參數變得更加容易。事實上，他們*今天*就可以更新他們的流程來新增額外的參數，為升級到 1.0 做準備。

另一個缺點是額外複製可能帶來的效能影響。但同樣，輕微的效能損失總比錯誤的流程行為要好。特別是當我們在前進的過程中，可以尋求其他方法來提高整個系統的效能。

在「以細微且難以察覺的方式破壞廣泛的流程」與「對更小的一個流程子集進行明顯的破壞」之間做選擇，我希望您能理解為什麼我們選擇了後者。

### 其他節點呢？

上述更改*僅*適用於 Function 節點。自定義節點仍負責在傳遞給 `node.send()` 之前根據需要複製訊息。

Node-RED 0.15 現在已經可以 [下載](https://github.com/node-red/node-red/releases/download/0.15.0/node-red-0.15.0.zip) 或透過 [npm 安裝](https://npmjs.org/package/node-red)。

如果升級，請閱讀 [升級說明](/docs/getting-started/upgrading_zh_TW.html)。

如果您使用的是 Raspberry Pi Jessie 上的預裝版本，儲存庫將在不久的將來更新。

---

### 編輯器更新


#### 調色板編輯器 (Palette editor)

現在可以在編輯器中管理您的調色板。這讓您可以瀏覽所有可用節點的目錄，並在不重啟 Node-RED 的情況下安裝其中任何一個。

![](/blog/content/images/2016/10/palette-editor.gif)

在底層，這會觸發 npm 將節點模組安裝到您的使用者目錄中 —— 預設為 `$HOME/.node-red`。請注意，如果節點除了標準 npm 依賴項之外還有任何依賴項，您需要手動解決它們。它還要求您安裝了 `npm`，如果您使用的是 Raspberry Pi 預裝映像檔，您可能沒有它 —— 請查看 [文件](/docs/hardware/raspberrypi_zh_TW.html#adding-nodes) 以了解如何獲取它。


#### 可捲動的標籤頁 (Scrollable tabs)

在上一個版本中讓您可以重新排列工作區標籤頁後，我們知道還有更多工作要做。一個常見的投訴是編輯器處理帶有大量標籤頁流程的方式；我們會將它們全部擠入視圖中，使得第一次就很難跳轉到正確的標籤頁。這也使得 Flows 選單隨著標籤頁超過頁面底部而變得無法使用。

在此版本中，我們的行為更加明智，允許標籤列捲動，因此標籤頁永遠不會變得太小。這確保了它們的標籤始終清晰可讀。我們也從 Flows 選單中移除了標籤頁列表。

![](/blog/content/images/2016/10/scrolling-tabs.gif)



但是等一下，我聽到你說，如果它們現在捲動出螢幕且沒有選單項，你該如何快速在標籤頁之間導航？我回答說，你想看看新的搜尋框...


#### 搜尋框 (Search box)

我們增加了一個搜尋框，可以輕鬆快速地找到任何標籤頁或節點。您可以從選單中的「Search flows」選項存取它，或者只需點擊 `Ctrl+.` (在 Mac 上為 `Cmd+.`)。當您輸入時，它會顯示可能的匹配列表，點擊您想要的那個，或者使用方向鍵並點擊 Enter。

如果您選擇了一個流程或子流程，它將切換到相應的標籤頁。如果您在流程中選擇了一個節點，它會切換到相應的標籤頁，捲動以確保節點可見，並閃爍其邊框以便您發現它。如果您選擇了一個配置節點，它將在配置節點側邊欄中顯示它。

![](/blog/content/images/2016/10/flow-search.gif)

#### 匯入/匯出改進

匯入和匯出對話框進行了一些小的更新。匯入流程時，您現在可以選擇將其匯入到新流程中，而不是目前流程。

匯出對話框為您提供更多關於匯出內容的選項 —— 無論是選定的節點、目前流程還是所有流程。您還可以選擇匯出內容應該是美化 (prettified) 還是壓縮 (minified)。

 ![](/blog/content/images/2016/10/import-dialog.png)
 ![](/blog/content/images/2016/10/export-dialog.png)

匯入時的其他幾個新功能：

 - 我們避免重複子流程，如果您匯入一個調色板中已有的子流程
 - 您現在可以在匯入的流程 JSON 中包含憑據 (credentials)


#### 其他事項

- 當您在流程之間切換時，我們會更新瀏覽器中的 URL，以便您可以直接鏈接到特定的流程或子流程：

    ![](/blog/content/images/2016/10/flow-url.png)


- 我們更改了新配置節點的預設範圍。當我們引入將配置節點範圍限定為特定流程的想法時，我們預設為您所在的目前流程。這引起了相當大的困惑，因為使用者不一定理解範圍設定在做什麼，最終在每個標籤頁上建立了相同的配置節點。在此版本中，我們現在將配置節點預設設置為全域 (global)。


### 運行時更新

#### 加密憑據 (Encrypting credentials)

我們一直謹慎確保任何憑據屬性不存儲在主流程文件中，並將它們放在專用的憑據文件中。這意味著主流程文件可以進行版本控制而無需擔心洩露敏感信息。然而，我們無法迴避的一個事實是，憑據確實需要與流程同步，僅對其中一個進行版本控制是沒有意義的。

由於我們正在建立對 Node-RED 的版本控制支持，我們需要做些什麼來保護憑據，同時使它們能夠被版本控制。

因此，我們現在預設對憑據進行加密。這將意味著您不再能手動編輯憑據文件，除非您選擇明確禁用加密。具體操作細節在 wiki 的 [這裡](https://github.com/node-red/node-red/wiki/Design%3A-Encryption-of-credentials)。


#### 防止覆蓋保護 (Preventing overwrite protection)

預期到版本控制支持，`/flows` 管理 API 已更新，在流程中包含了一個修訂版標識符。
當您點擊部署按鈕時，編輯器現在可以檢查運行時中的流程是否是它最初載入的那些。為什麼這有用？這意味著如果有人在您仍在處理您的更改時部署了一些更改，當您來部署時，編輯器會警告您。在此版本中，我們為您提供繼續部署（覆蓋已做的任何更改）或取消部署的選項。

我們已經開始致力於能夠查看已做了哪些更改，以便您在部署之前將其合併到您的更改中，但它沒有在這次發佈中趕上。

由於對 `/flows` 的更改不向後兼容，我們增加了一個新的自定義 HTTP 標頭來識別應該使用哪個版本的 API。這意味著任何現有的 `/flows` API 的使用將繼續有效，但可以輕鬆選擇加入新行為。此新 API 的文件將於本週晚些時候更新。

### Node 0.10

最後，我們需要談談 Node.js 0.10。在 [本月底](https://github.com/nodejs/LTS/blob/75bd36a060be3b965e696bc0c53f799089ad0f52/README.md)，它將不再接收 *任何* 更新 —— 包括安全更新。這種壽命終結 (end-of-life) 意味著我們很難證明維持對它的支持是合理的。已經有越來越多的模組掉落了對 0.10 和 0.12 的支持，如果發現關鍵問題，我們將陷入困境。

我們保留 0.10 這麼久的 *唯一* 原因是它是 Debian Stable 發行的版本 —— 因此它是我們 Raspberry Pi 預裝映像檔中唯一可用的版本。Debian Stable 還有一段時間才會更新，所以我們處於僵局。

在探索了廣泛的選項後，我們決定 0.15 系列將是最後一個支持 node 0.10 *和* 0.12 的系列。

**下一個里程碑版本 0.16 將需要 Node 4.x。**

這意味著，目前 Raspberry Pi 預裝版本將固定在 0.15。
為了讓生活更輕鬆，我們已經在預裝映像檔中包含了一個名為 `update-nodejs-and-nodered` 的腳本，它使用 NodeSource 儲存庫將 node.js 更新到最新的 LTS 版本，然後在上面重新安裝 Node-RED。有關更多信息，請參閱 [文件](/docs/hardware/raspberrypi_zh_TW.html#upgrading-nodejs) —— 包括根據需要手動執行該升級的指令。

#### Serial 節點

node.js 世界中最痛苦的部分之一是處理二進位依賴項。在版本之間升級以確保此類依賴項繼續如期運作可能是一件令人頭痛的事情。

我們正式發出通知，在 0.16 版本中 **我們將移除 `node-red-node-serialport` 作為預設依賴項**。

這意味著我們將不再在核心安裝中包含 Serial 節點。

這將移除許多硬體/平台相關的程式碼，加快安裝時間並縮小對於不需要該節點的使用者的運行時足跡。

這也將使升級更加可靠，因為序列埠程式碼將不再埋在 Node-RED 內部，可以像任何其他節點一樣進行升級。

這將意味著如果您正在使用序列埠節點，您需要在升級到 0.16 後手動安裝它們 —— 我們將在發佈該版本時分享更多細節。

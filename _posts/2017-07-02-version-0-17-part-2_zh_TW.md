我們在週五發佈了 Node-RED 0.17，並寫了一篇關於其中所有內容的常規部落格文章。幾個小時後，我們意識到漏掉了一些值得強調的內容。與其更新之前的文章，不如來個第二部分。

我們還發現了一些漏掉的小問題，因此隨同這篇文章，我們也發佈了 Node-RED 0.17.2。

---

### 先前在 0.17 中

#### 切換 Debug 狀態

Debug 節點在編輯器中一直有一個按鈕，讓您可以禁用其輸出。但這裡有一個隱藏的問題。節點是否被禁用存儲在流程配置中。在編輯器中點擊按鈕會在運行時切換該狀態，但不會保存它 —— 唯一的保存方式是點擊部署 (deploy) 按鈕。這是一個問題，因為如果您出於性能原因禁用了所有 Debug 節點，然後運行時由於某種原因重新啟動，它們將全部被重新啟用。我們知道這曾讓使用者措手不及並引發問題。

在 0.17 中，當您切換 Debug 狀態時，它仍然會立即生效，但 Deploy 按鈕會變為活動狀態，並且節點會被標記為已更改。這意味著您知道仍需要部署才能使更改永久保存。

#### 禁用流程

您現在可以在編輯器中禁用一個流程（即標籤頁），這意味著它仍保留在您的部署配置中，但其任何節點都不會啟動。這在您想暫時停止流程的某些部分而不刪除它們時特別有用。

要禁用一個流程，雙擊標籤頁以存取其屬性對話框。

在那裡，您還可以為流程添加描述。當您沒有選中任何節點時，這會顯示在資訊側邊欄中。對於為您的流程添加一些文件說明非常有用。

#### Join 節點的累積 (accumulate) 模式

我們提到了對 Split 節點所做的一些更改，但忽略了 Join 節點中的一個新模式。

Join 節點可用於使用序列中的每條訊息為該對象提供不同的鍵/值來構建對象 (Object) payload。在預設模式下，它會等待一組數量的訊息，然後傳遞構建的對象，然後再次從空對象開始。新模式將導致 Join 節點不重置為空對象，並會為每次後續更新發送構建的訊息。

如果您使用它來整理一組分別到達的傳感器值，並且希望在每次新值到達時始終發送最新的讀取值集，那麼這非常有用。

#### GPIO 引腳選擇

Raspberry Pi GPIO 節點的編輯器對話框以前僅公開了所有可用引腳的一個子集 —— 那些非雙重用途的引腳。在此版本中，所有的引腳現在都可供使用，並且對話框已更新以幫助識別哪個引腳是哪個。

#### JSON 格式化

JSON 解析器節點現在在將對象轉換為 JSON 時有一個美化輸出的選項。關於這一點沒什麼好說的了。

### 0.17.2 中的修復

 - 編輯器無法載入從具名範圍 (scoped) 模組載入的節點圖示 —— [issue #1305](https://github.com/node-red/node-red/issues/1305)
 - 如果您在編輯器中移動了一個節點並點擊部署，節點的更改標記（藍點）不會被清除 —— [issue #1310](https://github.com/node-red/node-red/issues/1310)
 - 將數據發送到 Debug 側邊欄已被改為與流程非同步，因為在重負載下它可能會導致意外的減速 —— [issue #1311](https://github.com/node-red/node-red/issues/1311)
 - GPIO 節點未設置為使用 BCM 模式

我們做得不太好的一件事是讓社群及時了解專案的整體進展。郵件列表和 Slack 團隊是持續不斷的討論和辯論，但很容易忘記並非每個人都關注所有不同的線索。

因此，我們將嘗試在此處進行更頻繁的發文，分享整個專案的進展情況。

### 🎉 100 萬次安裝 🎉

首先要強調的是，我們最近達到了 Node-RED 全球累計 100 萬次 npm 安裝。對於這個仍然讓我們感覺像是一個小專案的東西來說，這是一個令人難以置信的里程碑。

![](/blog/content/images/2018/03/node-red-npm-installs.png)

很難深入挖掘這個數字以了解更多關於 Node-RED 在何處以及如何被使用的資訊。我們可以肯定的一點是，大約 45% 的安裝發生在 [IBM Cloud](https://console.bluemix.net/catalog/starters/node-red-starter) 內 —— 我們可以從 `node-red-bluemix-nodes` 模組的安裝統計數據中推斷出這一點。它也不包括使用 Raspberry Pi 預裝版本的人，除非他們使用了我們的 [更新腳本](/docs/hardware/raspberrypi_zh_TW.html) 來獲取最新版本。

我們追蹤關於專案的各種其他統計數據，以了解社群是如何增長的，例如註冊郵件列表和 Slack 團隊的人數。雖然沒有任何一個數據本身是絕對指標，但它們都顯示出良好的穩定增長。


### 0.18 更新

我們對 0.18 中發佈的專案 (Projects) 功能收到了很多很棒的回饋，但仍有很多工作要做。

最常見的問題與遠端 git 儲存庫的身份驗證有關。還有一些懸而未決的問題需要解決 —— 最棘手的是與 Windows 上的 HTTPS 身份驗證有關。總的來說，如果您使用 SSH 金鑰，事情應該能正常運作。

我們也在思考專案功能的後續步驟。兩個熱門話題是：

1. 如何使專案成為可部署的產物。目標是能夠 clone 一個專案，執行 `npm install` 獲取其依賴項，然後只需 `npm start` 即可運行它 —— 而無需接觸編輯器。

2. 如何為您的節點設置環境特定的設置。例如，您可能希望在本地開發和測試時使用本地 MQTT 代理，但當您部署到「生產」環境時，您希望使用您的生產代理。我們已經支持使用環境變數來允許節點的動態配置（我們確實需要記錄這一點），我們希望看看如何透過「專案」更好地呈現這一點。

### 更新路線圖 (Roadmap)

1.0 路線圖仍然是我們工作的總體計劃，並將隨著進展不斷演變。我們排列一些技術內容的方式並沒有完全按計劃進行 —— 例如，庫 (Library) 的重新設計未能進入 0.18，我們目前對該特定部分沒有具體展望。

我們需要為 0.19 制定一個更具體的計劃，以免在下一個里程碑版本發佈前又過了 6 個月。最終版本的內容取決於社群的回饋以及所做的貢獻。

### 編輯器/運行時分離 (Editor/Runtime Split)

路線圖上最重要的功能之一是將編輯器和運行時拆分為各自的套件。這為 Node-RED 開啟了許多不同的使用案例。

這也是一個相當難以正確解決的問題；要確保我們在編輯器和運行時之間公開的 API 是正確的。這是我們過去幾週一直關注的項目。與所有主要功能項目一樣，有一個 [wiki 頁面](https://github.com/node-red/node-red/wiki/Design%3A-Runtime-Editor-Split)，我們正在那裡進行設計。目前紙上寫的內容比 wiki 上的多，但在接下來的幾天裡這種情況會改變。


### 測試節點

Mike Blackstock 在幫助標準化第三方如何為其節點編寫測試用例方面發揮了領導作用。核心節點已經有一個測試輔助模組，Mike 已將其提取到其自己的儲存庫 [node-red-node-test-helper](https://github.com/node-red/node-red-node-test-helper) 中。

這是一項正在進行中的工作，但希望能成為建立高質量節點的寶貴工具。

如果您想了解更多資訊，請到 [Slack](https://nodered.org/slack/) 的 #testing 頻道聊天。

### 流程庫 (Flow Library) 更新

Mike 最近也一直忙於為 [流程庫 (Flow Library)](https://flows.nodered.org) 進行一些急需的更新。

該頁面以前每次載入時都會檢索所有節點和流程的完整清單。在早期這不是問題，但現在我們列出了超過 2000 個項目，是時候讓初始頁面載入變得更有效率了。

感謝 Mike 的工作，它現在一次載入一頁項目，使載入速度快得多。這方面還有一些微調要做，但這是一個堅實的起點。

不僅如此，Mike 還增加了為任何節點評分（滿分 5 分）的功能 —— 如果您透過 GitHub 按鈕登入的話。隨著時間的推移，這將讓使用者更好地了解任何特定節點的質量。我們也將在某個時候研究如何在編輯器的調色板管理中呈現這些評分。

我們在每個節點的頁面上增加了一個「舉報此模組 (Report this module)」連結。如果您對節點的內容有任何疑慮，例如不適當或不合適的材料，這可以讓您通知管理員進行查看。這 *不是* 關於報告錯誤和尋求使用節點的幫助 —— 此類報告應直接提交給節點作者。


### 食譜 (Cookbook)

最後，一個呼籲。我們在一年前開始了食譜專案，旨在收集許多簡單的「食譜」，涵蓋您可以使用 Node-RED 做什麼的基礎知識。

隨著時間的推移，希望能將其擴展到涵蓋更複雜的食譜，但我們渴望確保首先涵蓋基礎知識。

遺憾的是，我們在填寫食譜方面的進展不如預期。

如果您是 Node-RED 的熱衷使用者並想回饋社群，這將是一個很好的參與方式。請到 [Slack](https://nodered.org/slack/) 的 #docs 頻道了解更多資訊。

我們的 [1.0 版路線圖](/blog/2017/07/17/roadmap-to-1-dot-0_zh_TW)於 2017 年 7 月發佈。它描述了我們想要在專案中解決的一系列主題，以及我們決定達成目標所需的各項功能。

從那以後，我們透過三個里程碑版本（包含其許多功能）在路線圖上取得了逐步進展。但仍有一些功能尚未完成，按照目前的進度和速度，還需要 6 個月才能完成。

隨著 0.20 版本的發佈以及執行環境/編輯器分離工作的完成，我們一直在討論如何更早地達成 1.0。我們一直在挑戰支撐路線圖的假設，看看它們是否仍然成立。

關鍵的體認是，1.0 並不是專案的終點。它只是另一個版本，儘管它理所當然地象徵著專案的成熟。

1.0 路線圖的主要概念是達到功能「完整性」和 API 穩定性。

### 功能「完整性」

這意味著我們已經解決了使用者向我們反饋的主要功能差距。這集中在較大的項目上 — 那些必須在核心執行環境中解決的部分，而不是作為對現有節點的增量增強。

專案 (Projects, 0.18)、可持久化 Context (0.19) 和子流程實例屬性 (0.20) 都符合這些標準。每一項都解決了非常特定的需求，並極大地增強了使用者可以使用 Node-RED 構建的內容。

### API 穩定性

每當我們向執行環境新增新功能時，它們都會引入新的 API 和擴展點。隨著越來越多的使用者希望將 Node-RED 與其現有應用程式整合，或者根據自己的需求進行自定義，確實需要一個穩定的 API。

執行環境/編輯器分離 (0.20) 的工作是這裡的主要項目 — 它暴露了 Node-RED 更多的內部組件，並提供了可以重複使用的 API。要正確記錄這些 API 並提供如何使用它們的具體範例，還有更多工作要做，但技術工作已經完成。

### 還剩下什麼？

原始路線圖中還有三個項目尚未交付：

 - 媒體庫重新設計 (Library Redesign) — 對內建媒體庫體驗的全面改革；這是 Node-RED 使用者體驗中最古老的部分之一，一直沒有得到太多關注。
 - 可插拔的訊息路由 (Pluggable Message Routing) — 能夠將自定義程式碼插入到訊息如何在節點之間傳遞的過程中。
 - 新的訊息 API (New Message API) — 對節點如何處理訊息的更新，這將使執行環境能夠更好地追蹤節點正在執行的操作，並啟用自動逾時節點等功能。

這三個項目的工作已經不同程度地展開，從高階設計工作到實際的程式碼貢獻。但它們都還有更多工作要做。

還有一些原始路線圖中未列出的活動線，包括：

 - 子流程屬性 UI — 允許子流程定義自己的自定義 UI 以設置其實例屬性
 - 整合式流程測試 (Integrated Flow Testing) — 能夠在編輯器中為您的流程定義測試案例，並擁有一個可用於驗證它們的測試運行器
 - 流程 Linter — 一個命令列工具，可用於根據一組關於流程應該做和不應該做的規則來驗證流程

我們面臨的風險是，隨著所有這些活動的進行，我們會在達成 1.0 之前不斷發現「還有一件事」要做。每個新項目都帶來了使現有功能不穩定的風險，並在計劃中增加了更多時間。


### 更改計劃

考慮到這一切，我們將進行一些重新優先排序，以確保我們只專注於 1.0 絕對必要的內容。

以下是這對剩餘路線圖功能的意義：

#### 媒體庫重新設計

這將為使用者在編輯器中與媒體庫的互動提供急需的外觀改進，但不會引入任何新功能。底層 API 將基本保持不變。

我們不會在 1.0 版本中增加插入其他媒體庫來源的能力，但 UI 變更會在未來考慮這一點。

#### 可插拔的訊息路由

此功能不再列入 1.0 的計劃中。就其本身而言，它不會為使用者增加任何價值。真正的價值來自於隨後可以插拔的事物 — 例如互動式流程除錯器。與其匆忙完成此項目的設計，不如在 1.0 之後完成它，屆時我們也可以並行開發一些具體的用途。

#### 新的訊息 API

我們已經達成了一段時間的設計共識，並且還有一個實作該設計的 pull request 正在等待中。但在過去幾週中，我們發現了一些擔憂，即新的 API 對於它將帶來的價值來說變動太大了。隨著 1900 多個節點模組的發佈，我們需要對在該領域引入的任何變更保持謹慎。目前正在討論一種替代設計，這是一種更適度的變更，提供了所需的大部分功能。

#### 非功能性變更

1.0 版本的發佈是一個在專案周圍引起轟動的機會。這將為社群帶來新使用者，因此我們需要確保擁有一切必要條件來引導他們使用 Node-RED 並幫助他們入門。

因此，除了 1.0 的技術工作外，我們還希望對網站進行清理，審查我們的文件並在食譜 (cookbook) 中增加更多內容。如果您正在尋找為專案做出貢獻的方法，但不確定是否要投入程式碼開發，那麼在這一領域有很多可以提供幫助的地方。

### 1.0 何時發佈？

我們的目標是讓 Node-RED 的*下一個*里程碑版本成為 1.0 版本。

目標是在 5 月中旬發佈第一個 1.0-beta 版本，並在 6 月初發佈正式版本。

### 然後呢？

正如我們在開始時所說，1.0 版本並不是專案的終點 — 它是專案成長的下一步。

在發佈的同時，我們將有一個清晰的持續工作路線圖，帶領我們走向下一個里程碑及以後。

我們還將有社群調查的結果回饋到計劃中 — 如果您還沒有完成調查，[請填寫](https://tiny.cc/NodeRED2019Survey)！

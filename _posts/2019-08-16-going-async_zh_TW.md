在 Node-RED 1.0 中，我們正在將節點之間傳遞訊息的方式從同步更改為非同步。在某些情況下，這將改變流程中處理訊息的相對順序。這篇部落格文章解釋了我們所說的同步和非同步是什麼意思、我們為什麼要進行這項更改以及它會產生什麼影響。

### Node.js 事件迴圈 (Event Loop)

要了解訊息如何通過 Node-RED 流程，我們首先需要稍微了解一下 Node.js 的運作方式。

由於 JavaScript 是一門單執行緒語言，它一次只能做一件事。但有時它需要執行一些耗時的工作，例如發起 HTTP 請求或將內容寫入檔案。如果它在其唯一的執行緒上執行這些工作，那麼它將阻塞其他任何事情的發生，效能將會非常糟糕。

相反，Node.js 將這些類型的操作交給底層作業系統（它是多執行緒的），並註冊一個回呼函數 (callback)，該函數將在操作完成時被呼叫。這就是 [Node.js 核心的事件迴圈 (Event Loop)](https://nodejs.org/en/docs/guides/event-loop-timers-and-nexttick/) 概念。

簡單來說，您可以將其想像成一個工作佇列。每當事件迴圈執行時，它會取出下一項工作並執行，而這又可能導致新的事件被添加到佇列中。


### 同步 vs 非同步

當我們談論同步和非同步程式碼時，您可以這樣理解：

 - 同步程式碼都在事件迴圈的一次執行中完成。
 - 非同步程式碼從事件迴圈中的一項工作開始，該工作隨後會將另一項工作添加到事件迴圈中，並在未來的執行中執行。

理解非同步程式碼的關鍵在於，一旦它向事件迴圈添加了更多工作，並且目前工作已「完成」，其他工作就可以執行。

### 同步訊息傳遞

自 Node-RED 啟動以來，我們一直在節點之間使用同步訊息傳遞。

這意味著當一個節點呼叫 `node.send(msg)` 時，該呼叫會傳遞到下一個節點的 `input` 事件處理程式，該處理程式執行其工作並呼叫下一個節點的事件處理程式，依此類推。如果每個節點的事件處理程式都是純同步程式碼，那麼訊息將在事件迴圈的一次執行中傳遞到流程的最末端。

![](/blog/content/images/2019/08/flow1.svg)

如果其中一個節點包含非同步程式碼（例如 HTTP Request 節點），那麼目前的執行將在該節點結束，下一項工作就可以開始。在這種情況下，如果第二條訊息的非同步工作比第一條訊息先完成，那麼它也有可能超越第一條訊息。

![](/blog/content/images/2019/08/flow1-node-async.svg)

### 流程分支

對於同步流程，當流程產生分支時，每個分支將依次完成。

![](/blog/content/images/2019/08/flow2.svg)

當您在流程的每個點添加 Debug 節點時，這確實會導致一些稍微違反直覺的行為。在下圖中，請注意訊息抵達 Debug 節點的順序。

### 更改為非同步訊息傳遞

在 Node-RED 1.0 中，我們將訊息傳遞更改為非同步。這意味著當一個節點呼叫 `node.send(msg)` 時，呼叫下一個節點的 `input` 事件處理程式的工作將被放入佇列中，以便在事件迴圈稍後的執行中呼叫。

對於那些更熟悉事件迴圈的人來說，我們使用的是 `setImmediate()`，因此它們實際上是在目前事件迴圈反覆運算的「檢查 (check)」階段被調用的。

回顧一下我們開始時的單分支、完全同步的流程，訊息現在將在流程中取得相等的進度。

![](/blog/content/images/2019/08/flow1-async.svg)

當流程產生分支時，分支將被「並行」評估。

![](/blog/content/images/2019/08/flow2-async.svg)

這也意味著在每個點都帶有 Debug 節點的流程將按照預期的順序記錄訊息。

![](/blog/content/images/2019/08/flow4-async.svg)

### 為什麼需要這項更改？

將訊息傳遞更改為非同步是出於多種原因。

#### 可插拔的訊息路由

路線圖上的其中一項功能（將在 1.0 之後推出）是能夠將自定義程式碼插入訊息路由路徑。該自定義程式碼可能需要執行非同步工作 — 例如在分散式 Node-RED 環境中通過網路發送訊息。

#### 節點逾時

我們正在研究執行環境如何更好地監控流經流程的訊息，並提供一種標準方法來為任何耗時過長的節點設定逾時。

在目前的同步模型中，節點處理訊息所需的時間是執行其自身程式碼所需的時間，加上後續每個節點處理傳遞給它的訊息所需的時間之和。

再次觀察這個分支流程，假設我們希望每個節點處理訊息的時間不超過 5 秒。

![](/blog/content/images/2019/08/flow2.svg)

如果一切都是同步的，那麼直到黃色和紅色訊息都抵達其 Debug 節點後，第二個節點才算「完成」。如果每個節點處理訊息需要 2 秒，那麼第二個節點處理其訊息將需要 10 秒，並且會發生逾時，儘管沒有單個節點的耗時超過 5 秒。

使用非同步訊息傳遞，一旦節點為下一個節點排隊了工作，我們就可以停止計時。

在 1.0 之後還有更多工作要做，以建立在這種能力之上，但轉向非同步是關鍵的第一步。

我們很快會發佈另一篇部落格文章，內容涵蓋 1.0 中節點訊息 API 的變更，以進一步支援這種逾時行為。

#### 更好的 I/O 排程

如果您還記得，事件迴圈的目的是允許 Node.js 執行環境在背景執行 I/O，並在有工作需要處理時進行回呼。

但只有在事件迴圈能夠取得定期進展的情況下，這些回呼才能發生。如果您有一大段同步程式碼，那麼您就是在阻止這些回呼被呼叫。這裡需要權衡：純同步程式碼會更快，但它會耗盡事件迴圈資源。

通過將大型同步流程拆分為較小的非同步區塊，它允許 Node.js 執行環境更好地排程執行環境中的所有其他活動。


### 這項更改會破壞我的流程嗎？

對於大多數流程來說，這項更改不會以任何方式改變其行為。我們一直表示，一旦流程產生分支，就不應對順序做出任何假設。

儘管如此，肯定有一些流程利用了觀察到的順序並做出了其中一些假設。因此，如果您知道您的流程做出了此類假設，則在升級時應多加註意。

這就是我們將這項更改作為 1.0 版本的一部分的原因。我們一直努力確保 Node-RED 在版本之間保持向下相容，但有時我們必須做出可能產生影響的更改。我們不會輕易做出這些更改，而且我們只能將其作為大版本發佈的一部分。


#### 暫時保持同步

鑑於這項更改可能會改變*某些*流程的行為，我們引入了一個新設定，該設定將恢復同步交付模式：

```
    runtimeSyncDelivery: true
```

鑑於路線圖中需要非同步交付的功能，此設定*不*是長期解決方案。請考慮其用法已立即被棄用。它僅旨在作為一項權宜之計，允許受影響的流程在更新以處理新的非同步模式之前升級到 1.0。

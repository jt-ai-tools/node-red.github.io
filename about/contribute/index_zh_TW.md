任何貢獻的起點都應該是與社群進行討論。我們首選的機制是專案論壇，儘管在 Slack 團隊上的討論同樣有效。這有兩個目的：

 - 它可以觸及廣大的受眾，包括那些不是 GitHub 使用者的人。
 - 它允許將實際上是支援請求或重複項目的功能請求在進一步處理之前進行過濾或合理化處理。

一旦社群討論達成結論，請求要麼被捨棄，要麼被推進。

對於來自社群成員且無法同時貢獻設計或程式碼的功能請求，提交者（Committer）會將該項目新增到 [Trello 白板](https://trello.com/b/R0O3CSrI/node-red-whiteboard)。白板作為功能積壓（backlog），在排定工作優先順序時可以參考。

如果一個項目有貢獻者（Contributor）來執行工作，下一步將取決於更改的性質。

如果它是一個影響有限且定義明確的功能——例如為現有節點新增新選項——應在適當的儲存庫中提出一個 Issue，並套用 `feature` 標籤。Issue 應提供功能的描述和所需的任何關鍵設計點。然後可以使用此 Issue 來協助完善提案。Issue 應清楚地識別誰正在處理它——這是為了避免重複工作並確保工作有一個核心焦點。

如果它是一個範圍較大的功能，可能需要多個 Pull Request 才能實現，或者對最終使用者有很大影響，則應在 [node-red/designs](https://github.com/node-red/designs) 儲存庫中建立設計說明。

一旦設計獲得批准並移至 `in-progress` 狀態，應在適當的儲存庫中提出一個 Issue，並套用 `feature` 標籤。如果設計為該功能確定了多個交付階段，則應根據需要為每個階段提出一個 Issue。Issue 應引用設計說明。如果計劃在特定版本中發佈，則每個 Issue 都應設定里程碑。

在某個時間點，Pull Request 將會送達。這將經歷正常的審查流程。提交者社群有責任及時審查 PR。對於任何重大更改，PR 應在清單中保留至少 48 小時——並考慮週末和其他假期。這讓所有提交者都有公平的機會審查 PR，或者至少登記他們進行適當審查的願望，無論他們身處哪個時區。

如果沒有未解決的異議，提交者將接受 PR。

如果有異議，應在提交者之間達成適當的共識。

一旦與 Issue 相關的所有 PR 都已解決，即可關閉 Issue。

一旦合併，相應的功能 Issue 應關閉。如果存在相應的設計說明，其歷史記錄部分應更新以反映該項目的交付地點/時間。


### Pull Request 要求

任何引入程式碼更改的 Pull Request 在被接受之前都預期要符合某些標準：

1. PR 的所有提交者必須簽署 JS 基金會 CLA。專案無法接受來自未知提交者的任何更改。

2. 這不應該是專案第一次知道這項貢獻。如上所述，任何程式碼貢獻都應該已經與社群討論過並提出了 Issue。

   如果 PR 包含錯誤修復，只要沒有爭議性的更改，它仍然可以被接受。

   如果是一個較大的更改，在遵循正確的貢獻流程之前，它可能會被拒絕。雖然我們不想阻礙貢獻，但我們確實希望避免跳過適當的審查/討論流程。

3. 建置測試應完全通過；它們會在每個 PR 上自動執行並回報結果。這涵蓋了程式碼風格、linting 以及單元和功能測試。

4. 更改必須包含適當的測試覆蓋率。核心專案的目標是維持 90% 以上的程式碼覆蓋率。如果 PR 導致程式碼覆蓋率下降，除非提供了充分的理由，否則它將被拒絕。

專案對在提出 PR 之前使用何種開發方法進行程式碼更改不作要求。專案高度重視良好的測試覆蓋率和文件。

### Git 工作流程

主要專案儲存庫在 Git 中使用以下工作流程。

與傳統的 Git 工作流程一樣，不同的活動流使用多個分支。以下描述了我們目前的使用方式。

 - `master` - 此分支包含 Node-RED 最新發佈的版本，以及自那時以來所做的任何其他錯誤修復。在任何時候，此分支都可以作為下一個維護版本發佈。在撰寫本文時，這包含 0.16.2 和一些將在某個時候作為 0.16.3 發佈的修復。
 - `dev` - 此分支包含下一個里程碑版本的開發工作。這是開發新功能的地方。
 - 較大的功能在各自的分支中開發，並在功能足夠穩定以供更廣泛使用時合併到開發分支。

### 工具

專案使用多種工具來支援其活動。以下清單是目前使用情況及其職能的反映。


 - [論壇](https://discourse.nodered.org)

   這是專案的主要討論場所。任何人都可以加入論壇。

 - [Slack](/slack_zh_TW.html)

   對於活躍的即時對話，Slack 可能比論壇更有成效。任何人都可以加入 Slack 團隊。`#dev` 頻道用於討論專案開發。

 - [Trello - 白板](https://trello.com/b/R0O3CSrI/node-red-whiteboard)

   對想法和功能的高階追蹤。這是將被視為現有功能的增強或大型「史詩級」（epic）項目的項目捕獲到積壓中的地方。

   TC（技術委員會）使用它來提供跨項目的優先順序感。任何人都可以查看看板。提交者具有寫入權限。

 - [Trello - 文件](https://trello.com/b/m2mBMUYj/documentation)

   一個獨立的看板用於處理與特定程式碼更改無關的文件任務。這有助於確保我們為不同類型的使用者/開發者/貢獻者提供正確的文件。任何人都可以查看看板。提交者具有寫入權限。

 - [GitHub - 程式碼](https://github.com/node-red/node-red)

   專案原始碼的版本控制。任何人都可以複製或 fork 儲存庫；沒有私有儲存庫。提交者可以將提交套用到儲存庫。

 - [GitHub - Issue](https://github.com/node-red/node-red/issues)

   錯誤/Issue 的回報以及功能開發的追蹤。任何人都可以回報 Issue。提交者可以「擁有」一個 Issue。

 - [GitHub - Pull Request](https://github.com/node-red/node-red/pulls)

   如何對儲存庫進行任何貢獻。任何人都可以提出 Pull Request。具體流程在別處另有說明。

 - [GitHub - Wiki](https://github.com/node-red/node-red/wiki)

   每個儲存庫都有一個與之關聯的 Wiki。它們被用作以協作方式開發設計、想法和文件的工室。

   特別是，核心儲存庫 Wiki 有一組 [設計說明](https://github.com/node-red/node-red/wiki/Design-Notes) 頁面。這是可以開發設計並尋求回饋的地方。

   Wiki 沒有存取控制；任何人都可以進行更改。但應僅與正在積極處理頁面的人員協作進行更改。
